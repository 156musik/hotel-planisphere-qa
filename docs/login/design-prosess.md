# Login Screen Design Process

## Overview

本ドキュメントは、Hotel Planisphere のログイン画面を対象に実施したテスト設計プロセスをまとめたものです。

テスト観点の洗い出しから、テストケース作成、Playwrightによる自動化までの検討内容を記録しています。

---

## Design Flow

以下の流れでテスト設計を実施しました。

1. ログイン画面の仕様確認
2. テスト観点の洗い出し
3. テストケース作成
4. Playwrightによるテストコード実装
5. AIによるレビューおよび改善

---

## Test Points

### Test Case Selectionest Case Selection Rationale Rationale

本プロジェクトでは、すべての組み合わせを自動化するのではなく、E2Eテストとして優先度の高いシナリオを選定しました。

#### Normal Scenarios

正常系では、一般会員およびプレミアム会員の両方でログインできることを確認対象としました。

会員種別によって利用可能な機能や表示内容が異なるため、認証処理についても各会員種別で正常にログインできることを確認する必要があると判断しました。

また、実務において会員ステータスごとにログイン可否が異なってしまう不具合を経験したことがあり、その経験も考慮してテストケースを選定しています。

#### Negative Scenarios

異常系では、ユーザーが遭遇しやすい入力ミスを優先してテストケースを選定しました。

対象としたケースは以下のとおりです。

* メールアドレス誤り
* パスワード誤り

ログイン失敗の原因として、認証情報の入力ミスは発生頻度が高いと考えられるため、E2Eテストで優先的に確認しています。

なお、未入力やメールアドレス形式不正などの入力バリデーションについては、テスト観点・テストケースとして整理した上で、自動化対象を段階的に拡張する予定です. 

### Authentication

* 一般会員がログインできること
* プレミアム会員がログインできること
* ログアウトできること

### Input Validation

* メールアドレス未入力
* パスワード未入力
* メールアドレス形式不正
* 認証情報誤り

### UI

* エラーメッセージが表示されること
* エラーメッセージの内容が正しいこと
* ログイン成功後にマイページへ遷移すること

## Playwright Implementation

Playwright Code Generatorを利用して初期コードを生成し、その後以下の点を見直しました。

* RoleベースのLocatorを優先
* 不要なLocatorの削除
* 可読性を考慮したコード整理
* Assertionの見直し
* 正規表現を利用したテキスト取得

実装時には、LocatorやAssertionの動作確認を繰り返しながら改善を行いました。

---

## AI Usage

本画面の設計・実装では、生成AIを補助的に活用しました。

利用内容

* テスト観点のレビュー
* テストケースのレビュー
* Playwright実装時の技術相談
* Locator・Assertionの改善案
* READMEおよびディレクトリ構成のレビュー

AIが提案した内容はそのまま採用せず、実際の画面仕様および動作を確認した上で採否を判断しています。

---

## Lessons Learned

本画面の作成を通して以下を学びました。

* テスト観点からテストケースへ落とし込む流れ
* PlaywrightによるE2Eテストの基本的な実装
* Locatorの選定と保守性を考慮した記述方法
* GitHubによるソースコード管理
* GitHub Actionsによる自動実行環境の構築
