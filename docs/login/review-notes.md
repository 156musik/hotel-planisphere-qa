## 総評

「入力」カテゴリはメール形式バリデーション・パスワード文字種/文字数・スペース等の異常系まで詳細に網羅されており、完成度が高い。一方、**ログイン成功後の遷移**・**ログイン済みユーザーのリダイレクト**・**認証失敗エラーメッセージの文言と表示箇所**という、E2Eとして最も確認価値が高い観点が揃って欠落している。また、ログイン処理が他画面（index / plans）のナビゲーションに与える回帰影響も未カバーのため、テストケース化する前に優先的に補強が必要。

## カテゴリ別カバー状況

凡例: ○ 十分カバーされている　△ 一部あり・補強余地あり　× ほぼなし　― 対象外

| カテゴリ | カバー状況 | コメント |
| --- | --- | --- |
| 表示 | △ | ヘッダー・メイン・フッターの静的要素は充実。SP版でh1タイトルが非表示になる点と、ログイン済みでもナビ表示が変化しないこと（固定ナビの確認）が未カバー |
| 入力 | ○ | メール形式・パスワード文字種/文字数・スペース含む入力まで詳細に網羅されている |
| 認証・権限 | △ | 正常/異常ログインの組み合わせはカバー済み。ログイン済みユーザーがURLを直打ちした場合のリダイレクトが完全に欠落 |
| 遷移 | × | ログイン成功後のマイページ遷移が観点として存在しない。ナビ内リンクの遷移先確認もなし |
| エラー | △ | 入力・認証カテゴリに「挙動」として散在しているが、エラーメッセージの文言・表示箇所（2箇所）を独立した観点として持っていない |
| 非同期・ローディング | △ | 観点なし。クライアントサイドJSで認証が同期的に完結している可能性が高いため保留相当だが、要確認 |
| レスポンシブ | △ | PC/SP共通のレイアウト確認は含まれる。SP版でのh1タイトル非表示・ハンバーガーメニュー展開後の項目確認が弱い |
| 回帰影響 | × | ログイン成功が index.html / plans.html のナビゲーション切替に与える影響が一切未カバー |
| データ・境界状態 | △ | パスワード文字数の境界値は含まれる。メールアドレスの最大文字数が未カバー |

## 追加候補

### 採用推奨

```
[遷移] ログイン成功後、マイページ画面（mypage.html）へ遷移すること
根拠: tests/login/test-login.spec.ts:14（toHaveURL(/mypage/)）
視点: ユーザー視点QA
```

```
[認証・権限] ログイン済みユーザーがログインページのURLを直接入力した場合、トップページへリダイレクトされること
根拠: docs/login/research/login-analysis.md:80（src/login.js:5-8 の redirectToTop()）
視点: 権限・状態差分QA
```

```
[エラー] 認証失敗時のエラーメッセージ「メールアドレス、またはパスワードが違います。」がメールアドレス・パスワード各入力欄の直下に1件ずつ（計2箇所）表示されること
根拠: tests/login/test-login.spec.ts:48-51（toHaveCount(2)）
視点: ユーザー視点QA
```

```
[回帰影響] ログイン成功後、index.html / plans.html のヘッダーナビゲーションが「会員登録・ログイン表示」から「マイページ・ログアウト表示」へ切り替わること
根拠: docs/login/research/login-analysis.md:38-41、docs/common/research/research-header-footer.md:44-49
視点: 回帰影響QA
```

```
[遷移] ログイン後にログアウトボタンを押下すると、トップページ（index.html）へ遷移すること
根拠: docs/common/research/research-header-footer.md:51（action="./index.html"）
視点: ユーザー視点QA
```

```

根拠: docs/common/research/research-header-footer.md:13（class="d-none d-lg-block"）、同:26-28

```

### 要確認

```
[エラー] 未入力状態でログインボタンを押下した場合のエラーメッセージの文言・表示位置（認証エラーと同一文言か、必須入力エラーになるか）
確認先: ソースコードのバリデーション実装または実画面
視点: 仕様追及型QA
```

```
[認証・権限] 退会済みユーザー・ロックアウトユーザーでログインした場合のエラーメッセージの有無と文言
確認先: ソースコードまたは実画面（テストデータの入手方法も合わせて確認）
視点: 仕様追及型QA
```

```
[データ・境界状態] メールアドレス入力欄の最大文字数制限の有無
確認先: src/ 配下のバリデーション実装
視点: 異常系・境界値QA
```

```
[表示] ログイン済みの状態でログインページのナビゲーションが変化しないこと（login.htmlは固定ナビのため、JSによる切替が発生しないことを確認）
確認先: docs/login/research/login-analysis.md:72-87（グループBの仕様）
視点: 仕様追及型QA / 権限・状態差分QA
```

---

### 保留

```
[非同期・ローディング] ログイン処理中のローディング表示
保留理由: クライアントサイドJSで認証が同期的に完結している可能性が高く、非同期UIが存在するか不明
視点: 自動化適性QA
```

```
[入力] パスワード入力欄がマスク表示（●）になっていること
保留理由: ブラウザのinput[type="password"]デフォルト動作であり、E2Eより目視確認またはユニットテスト向き
視点: 自動化適性QA
```

```
[データ・境界状態] SQLインジェクション・スクリプト文字列（<script>等）の入力挙動
保留理由: クライアントサイドJSのみの構成でDBが存在せず、セキュリティテストとして別スコープで検討すべき
視点: 異常系・境界値QA
```

## 仕様確認が必要な点

- 未入力時のエラーメッセージ文言（認証エラー「メールアドレス、またはパスワードが違います。」と同一か、別の必須エラーメッセージが出るか）
- 退会済みユーザー・ロックアウトユーザーのエラーメッセージ内容と、テストデータの入手方法
- パスワードの具体的な最小・最大文字数制限値
- メールアドレス入力欄の最大文字数制限の有無

---

## 優先的に見直すべき観点

1. **ログイン成功後のマイページ遷移確認** ― 正常系の最終ゴールにあたるが、現在のCSVには「遷移」カテゴリの観点が1件もなく、テストケース化した際に抜け落ちるリスクが高い
2. **ログイン済みユーザーによるURLダイレクトアクセス時のリダイレクト** ― `src/login.js` に実装済みのアクセス制御だが観点が完全に欠落している。セキュリティ観点からもリスクが高い
3. **認証失敗エラーメッセージの文言・表示箇所（2箇所）** ― テストスペックで確認済みの仕様だが、観点として独立して存在しないためテストケース作成時に見落としが起きやすい
4. **ログイン後のindex / plansナビゲーション変化（回帰影響）** ― ログイン処理が他画面の表示に与える影響は `login-analysis.md` で確認済みだが現在の観点ファイルに一切含まれていない
5. **SP版h1タイトルの非表示確認** ― `d-none d-lg-block` というCSS仕様は `research-header-footer.md` で確認済みだが、現在の「サイトタイトルが表示されている」観点はPC前提でSPの非表示を見逃す可能性がある
