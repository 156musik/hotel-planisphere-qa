# テスト観点レビュー — ログイン画面

対象ファイル: `docs/login/test_perspectives_login.csv`
レビュー日: 2026-06-28

---

## 総評

「表示」「入力」「認証」の3カテゴリは詳細に洗い出されており、メールアドレス形式バリデーションの網羅度は特に高い。一方、**ログイン成功後の遷移**・**ログイン済みでのアクセス制御（リダイレクト）**・**ログイン後のナビゲーション変化**という、E2Eとして最も確認価値が高い観点が不足している。エラーメッセージについても「挙動」として入力・認証カテゴリに混在しており、「エラー」カテゴリの独立した観点が薄い。

## カテゴリ別カバー状況

凡例: ○ 十分カバーされている　△ 一部あり・補強余地あり　× ほぼなし　― 対象外

| カテゴリ | カバー状況 | コメント |
| --- | --- | --- |
| 表示 | △ | ヘッダー/メイン/フッターは充実。ただしログイン済み時のナビ表示差分（会員登録・ログインが非表示になること）は未カバー |
| 入力 | ○ | メール形式・パスワード文字種/文字数・スペースなど詳細に網羅されている |
| 認証・権限 | △ | 正常/異常ログインはカバー済み。ログイン済みユーザーがURLを直打ちしてログインページにアクセスした場合のリダイレクトが未カバー |
| 遷移 | × | ログイン成功後のマイページ遷移、ログアウト後のトップページ遷移が観点として明示されていない |
| エラー | △ | 認証エラーの発生は入力・認証カテゴリに散在しているが、エラーメッセージの文言・表示位置・表示タイミングを独立した観点として持っていない |
| 非同期・ローディング | × | 観点なし（このアプリはクライアントサイドJSのため該当する非同期処理があるか要確認） |
| レスポンシブ | △ | PC/SP両対応のレイアウト確認は含まれる。SP版ハンバーガーメニュー展開後の項目表示確認は観点として弱い |
| 回帰影響 | × | ログイン成功がindex.html / plans.htmlのナビゲーション切替に与える影響が未カバー |
| データ・境界状態 | △ | パスワード文字数の境界値は含まれる。メールアドレスの最大文字数は言及なし |

## 追加候補

### 採用推奨

- `[遷移]` ログイン成功後、マイページ画面（`mypage.html`）へ遷移すること ― 根拠: `tests/login/test-login.spec.ts:14`、`docs/login/design-process.md:68`
- `[認証・権限]` ログイン済みユーザーがログインページ（`login.html`）のURLを直接入力した場合、トップページへリダイレクトされること ― 根拠: `docs/login/login-analysis.md:80`（`src/login.js:5-8`）
- `[遷移]` ログアウト後、トップページ（`index.html`）へ遷移すること ― 根拠: `docs/login/login-analysis.md:64`（`action="./index.html"`）
- `[回帰影響]` ログイン成功後、index.html / plans.html のヘッダーナビゲーションが「会員登録・ログイン」から「マイページ・ログアウト」へ切り替わること ― 根拠: `docs/login/login-analysis.md:38-41`
- `[エラー]` 認証失敗時のエラーメッセージが「メールアドレス、またはパスワードが違います。」という文言でメール/パスワード各入力欄の下に表示されること ― 根拠: `tests/login/test-login.spec.ts:49`、`docs/login/test_cases_login.csv:30-33`

### 要確認

- `[エラー]` パスワード未入力時のエラーメッセージの文言と表示位置 ― 確認先: ソースコードまたは実画面（`test_cases_login.csv` 行22 補足に「仕様未定義のため推測」と記載あり）
- `[認証・権限]` 退会済みユーザーでログインした場合のエラーメッセージの有無と文言 ― 確認先: ソースコードまたは実画面（`test_cases_login.csv` 行34 補足に「エラーメッセージの内容は未定義」と記載あり）
- `[認証・権限]` ロックアウトされたユーザーでログインした場合のエラーメッセージの有無と文言 ― 確認先: ソースコードまたは実画面（`test_cases_login.csv` 行35 補足に「未定義」と記載あり）
- `[入力]` パスワードの具体的な最小・最大文字数制限 ― 確認先: `src/` 配下のバリデーション実装
- `[データ・境界状態]` メールアドレスの最大文字数制限の有無 ― 確認先: ソースコードまたは実画面

### 保留

- `[非同期・ローディング]` ログイン処理中のローディング表示 ― 保留理由: クライアントサイドJSで認証処理が同期的に完結している可能性が高く、非同期UIが存在するか不明
- `[入力]` パスワード入力欄がマスク表示（●）であること ― 保留理由: ブラウザデフォルト動作であり、E2Eより目視確認またはユニットテスト向き
- `[データ・境界状態]` SQLインジェクション・スクリプト文字列（`<script>`等）の入力挙動 ― 保留理由: クライアントサイドJSのみの構成であればDBが存在せず、セキュリティテストとして別スコープで検討すべき

## 仕様確認が必要な点

- パスワード未入力時のエラーメッセージの文言（認証エラーと同一か、必須エラーになるか）
- 退会済みユーザー・ロックアウトユーザーのエラーメッセージ内容と、そのユーザーアカウントの入手方法（テストデータ）
- パスワードの文字数制限（最小値・最大値）
- メールアドレス入力欄の最大文字数制限の有無

## 優先的に見直すべき観点

1. **ログイン済みユーザーによるログインページ直接アクセス時のリダイレクト** ― `login-analysis.md` で実装確認済みのアクセス制御観点だが現在の観点ファイルに完全に欠落しており、リスクが高い
2. **ログイン成功後のマイページへの遷移確認** ― 正常系の最終ゴールに相当するが「認証」カテゴリに埋もれており、「遷移」として独立した観点が必要
3. **エラーメッセージの文言・表示位置** ― 入力と認証の各観点に「挙動」として散在しているが、「エラーメッセージが正しい文言で正しい位置に出ること」という観点が独立して存在しないため、テストケース作成時に見落としが起きやすい
4. **ログイン後のナビゲーション変化（回帰影響）** ― ログイン処理がindex.html/plans.htmlの表示に影響することは `login-analysis.md` で確認済みだが観点として未カバー
5. **ログアウト後のトップページ遷移** ― ログイン→マイページ確認→ログアウトの一連フローを観点として持つことで、セッションのリセットまで含めた完全な正常系シナリオを担保できる
