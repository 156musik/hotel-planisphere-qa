# テスト観点レビュー

対象の画面・機能のテスト観点を読み込み、抜け漏れを指摘し、追加候補を分類して出力する。

## 前提

- 仕様として確認できていない挙動は断定しない。「〜と思われる」「要確認」などの表現を使う
- コードや調査ドキュメントが `docs/` にある場合は参照して根拠を示す
- 追加観点は以下の3つに分類する
  - **採用推奨**: 仕様・コードから明確に裏付けられる、テストすべき観点
  - **要確認**: 仕様が不明確、または実装を確認しないと判断できない観点
  - **保留**: 優先度が低い、または対象外の可能性がある観点

## チェックする観点カテゴリ

レビュー時は以下の8カテゴリを網羅的に確認する。

| カテゴリ | 確認内容の例 |
| --- | --- |
| 表示 | 初期表示の内容・ラベル・件数・フォーマット・空状態の表示 |
| 入力 | 入力制限・バリデーション・必須/任意・プレースホルダー・文字数 |
| 認証・権限 | ログイン状態による表示差分・未認証アクセス時のリダイレクト・ランク別表示 |
| 遷移 | リンク先URL・新タブ/同タブ・遷移後の状態・戻り遷移 |
| エラー | エラーメッセージの内容・表示タイミング・複数エラーの扱い |
| レスポンシブ | SP/タブレット/PCでのレイアウト差分・非表示要素・列数変化 |
| 回帰影響 | 変更が他画面のナビ・共通コンポーネントに与える影響 |
| 仕様不明点 | コードに存在するが仕様書に記載がない挙動・条件分岐 |

## 手順

1. ユーザーが提示したテスト観点（CSV・箇条書き・表など）を読み込む
2. 対象画面の調査ドキュメント（`docs/` 配下）があれば参照する
3. 上記8カテゴリに照らして観点の網羅性を確認する
4. 以下の形式で出力する

## 出力形式

### 総評

現在の観点の網羅度・質について2〜3文で評価する。強みと課題を両方述べる。

---

### 不足している観点

現在の観点に存在しない、またはカバーが薄いカテゴリと具体的な内容を列挙する。

- **[カテゴリ名]**: （具体的な不足内容）
- ...

---

### 追加候補

#### 採用推奨

仕様・コード・調査ドキュメントから根拠が確認できる観点。コード根拠があれば併記する。

- （観点の内容）　※根拠: `ファイル名:行番号`
- ...

#### 要確認

仕様が不明、または実装を確認しないと判断できない観点。

- （観点の内容）　※確認先: （仕様書・コード・担当者など）
- ...

#### 保留

優先度が低い、スコープ外の可能性がある、または現時点では判断できない観点。

- （観点の内容）　※保留理由: （理由）
- ...

---

### 仕様確認が必要な点

テスト設計を進める前に仕様を確認すべき事項をまとめる。

- （確認事項）
- ...

---

### 優先的に見直すべき観点

リスクが高い・影響範囲が広い・現在カバーが薄いなどの理由で、特に注力すべき観点を3〜5件挙げる。

1. （観点と理由）
2. ...
