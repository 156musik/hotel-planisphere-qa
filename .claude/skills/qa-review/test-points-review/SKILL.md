# テスト観点レビュー

## 呼び出し方

```
/test-points-review <観点ファイルのパス> [画面名]
```

例:
- `/test-points-review docs/login/test_perspectives_login.csv ログイン画面`
- `/test-points-review docs/plans/ プラン一覧画面`

画面名を省略した場合はファイルパスから推測する。引数がない場合はユーザーに確認する。

---

## 方針

- **既存の観点を否定しない。** 不足している観点を補完することが目的。
- **仕様として確認できていない挙動は断定しない。** 「〜の可能性がある」「要確認」などの表現を使う。
- **E2Eテストとして優先度の高い観点を重視する。** 細粒度のロジック検証はユニットテストで担保すべき観点として「保留」に分類する。
- **根拠を示す。** `docs/` の調査ドキュメントまたはソースコードが根拠になる場合はファイル名・行番号を明記する。

### 追加観点の分類基準

| 分類 | 判断基準 |
| --- | --- |
| **採用推奨** | `docs/` またはソースコードで仕様・実装が確認でき、E2Eで検証すべきと判断できる |
| **要確認** | 仕様が不明確、または実装を確認しないと判断できない。テスト設計を進める前に確認が必要 |
| **保留** | 優先度が低い・スコープ外の可能性がある・ユニットテストで担保すべき・現時点では判断不能 |

## 手順

### Step 1. 観点ファイルを読み込む

引数で指定されたパスのファイル（CSV・Markdown・テキスト等）を Read で読み込む。
ディレクトリが指定された場合はその配下のファイルをすべて読む。
読み込めない場合はユーザーに確認する。

### Step 2. 関連ドキュメントを探索する

以下の優先順で対象画面に関連するドキュメントを探索し、Read で読み込む。

1. `docs/` 配下のサブディレクトリを確認する（`ls docs/`）
2. 画面名に対応するディレクトリ・ファイルを読む（例: ログイン画面なら `docs/login/` 配下）
3. `tests/` 配下に対応するスペックファイルがあれば読む
4. 参照すべきソースコードがある場合は必要な箇所のみ読む

ドキュメントが見つからない場合は、その旨を出力に記載してレビューを続ける。

### Step 3. カテゴリ別に網羅性を確認する

以下の9カテゴリと既存観点を照合し、カバー状況を判定する。

| # | カテゴリ | E2Eで優先すべき確認内容の例 |
| --- | --- | --- |
| 1 | **表示** | 初期表示の内容・ラベル・件数・フォーマット・空状態・ローカライズ |
| 2 | **入力** | 必須/任意・バリデーション・文字数制限・入力形式（メール・数値等）|
| 3 | **認証・権限** | ログイン状態による表示差分・未認証時のリダイレクト・ランク別表示制御 |
| 4 | **遷移** | リンク先URL・新タブ/同タブ・遷移後の状態・パラメータの引き継ぎ |
| 5 | **エラー** | エラーメッセージの文言・表示タイミング・複数エラーの扱い・回復操作 |
| 6 | **非同期・ローディング** | スピナーの表示/消去・データ取得後の描画・エラー応答時の表示 |
| 7 | **レスポンシブ** | SP/タブレット/PCでのレイアウト差分・非表示要素・列数・ハンバーガーメニュー |
| 8 | **回帰影響** | 変更が他画面のナビ・共通コンポーネント・ヘッダー/フッターに与える影響 |
| 9 | **データ・境界状態** | 0件/最大件数/長い文字列/特殊文字などの境界データでの表示確認 |

画面タイプによって適用しないカテゴリがある場合（例: 入力フォームのない一覧画面では「入力」カテゴリはスキップ）は、カバー状況の欄に「対象外」と記載する。

### Step 4. 出力を生成する

下記「出力形式」の構造に従って出力する。既存観点のリスト自体は出力に含めない。

## 出力形式

以下の構造で出力すること。

---

### 総評

現在の観点の網羅度・強みと主な課題を2〜3文で評価する。

---

### カテゴリ別カバー状況

凡例: ○ 十分カバーされている　△ 一部あり・補強余地あり　× ほぼなし　― 対象外

| カテゴリ | カバー状況 | コメント |
| --- | --- | --- |
| 表示 | ○ / △ / × / ― | （補足があれば） |
| 入力 | ○ / △ / × / ― | |
| 認証・権限 | ○ / △ / × / ― | |
| 遷移 | ○ / △ / × / ― | |
| エラー | ○ / △ / × / ― | |
| 非同期・ローディング | ○ / △ / × / ― | |
| レスポンシブ | ○ / △ / × / ― | |
| 回帰影響 | ○ / △ / × / ― | |
| データ・境界状態 | ○ / △ / × / ― | |

### 追加候補

#### 採用推奨

`docs/` またはソースコードで根拠が確認できる観点。

- `[カテゴリ名]` （観点の内容） ― 根拠: `ファイル名:行番号`
- ...

#### 要確認

テスト設計を進める前に仕様や実装を確認すべき観点。

- `[カテゴリ名]` （観点の内容） ― 確認先: （仕様書・コード・担当者など）
- ...

#### 保留

優先度が低い・スコープ外・ユニットテスト向き・現時点で判断不能な観点。

- `[カテゴリ名]` （観点の内容） ― 保留理由: （理由）
- ...

---

### 仕様確認が必要な点

テスト設計を進める前に確認すべき仕様上の疑問点をまとめる。

- （確認事項）
- ...

「要確認」の観点と内容が重複する場合は省略してよい。

---

### 優先的に見直すべき観点

リスクが高い・影響範囲が広い・現在カバーが薄いなどの理由で、特に注力すべき観点を3〜5件挙げる。理由も1行で添える。

1. （観点） ― （優先する理由）
2. ...
